# 工地塔吊占用争议处理逻辑说明

## 一、问题背景

现在在考虑做审批的一种情况，目标场景是用于工地塔吊的申请，但是申请存在占用驳回的情况，这种时候就需要管理员介入决定是归属谁来占用塔吊。

举个例子：张三在七月五号上午十点申请了一条塔吊 A 的中午十二点至一点的占用，产生了一条占用表单【TC-RWS-20260703154523】，然后李四现在对这个时间的占用发起争议表示自己也要使用这个时间段，产生了一条争议单【D-RWS-20260704171345】，这条争议单会通知给管理员王五，由王五根据争议单【D-RWS-20260704171345】来决定这个时间段到底归张三还是李四，这个简单的逻辑没有问题。

现在的问题点是，如果还是刚刚的例子情况，在管理员王五做出决定之前，孙六也对占用表单【TC-RWS-20260703154523】发起了争议，产生了争议单【D-RWS-20260704171525】。注意这个情况下两条争议单其实都是基于初始的占用表单发起的，管理员王五会在张三、李四、孙六三个人里选一个决定最终的归属。

## 二、待确认的核心问题

### 问题①

如果王五同时收到了两条，先把李四的处理了，是不是这条占用表单所有相关的争议单都应该是从待处理变成已处理状态？

### 问题②

如果王五在先收到李四的争议单处理完变成待处理状态后，又收到了孙六的争议单，那是不是李四的争议单状态为已处理但是孙六的为待处理，然后等孙六的争议单处理完后再变成已处理？

### 问题③

如果还有另外一个管理员周七也看到了这条 todo，孙六处理时选择了张三，周七选择了孙六，那最终结果是以最终结算时间为准吗？

### 问题④

如果李四发起争议后，张三把原本的占用单取消了，这里如何处理？是限制发起争议后不让占用单点击取消，还是说直接把张三的占用直接改为李四（因为张三放弃这个时间段了），或者还有一种情况，张三李四孙六都是参与人员，张三取消后，争议单里只用去处理李四和孙六的争议？

## 三、建议方案

建议把这个逻辑拆成两层：**争议单是记录，管理员真正处理的是“占用冲突事件”**。否则多个争议单会把状态、待办、最终归属搞乱。

## 四、核心建议

管理员待办列表不要按每一张争议单独立处理，而是按「同一塔吊 + 同一时间段 + 同一初始占用单」聚合成一条待办。

也就是说：

| 层级 | 作用 |
|---|---|
| 占用表单 | 张三原始申请塔吊 A 的 12:00-13:00 |
| 争议单 | 李四、孙六分别表达“我也要这个时间段” |
| 冲突事件 / 争议组 | 管理员最终裁定这个时间段归谁 |

你的例子里，王五看到的最好不是两条独立待办，而是一条：

| 待办类型 | 关联占用单 | 塔吊 | 时间段 | 当前占用人 | 争议人数 | 状态 |
|---|---|---|---|---|---:|---|
| 塔吊占用争议 | TC-RWS-20260703154523 | 塔吊 A | 7月5日 12:00-13:00 | 张三 | 2 | 待处理 |

点进去后展示：

| 申请人 / 争议人 | 来源单号 | 申请时间 | 诉求 |
|---|---|---|---|
| 张三 | TC-RWS-20260703154523 | 7月5日 10:00 | 原始占用人 |
| 李四 | D-RWS-20260704171345 | 7月4日 17:13 | 争议占用 |
| 孙六 | D-RWS-20260704171525 | 7月4日 17:15 | 争议占用 |

管理员只做一次决策：**最终归属张三 / 李四 / 孙六中的一人**。

## 五、问题①处理建议

**问题①：王五同时收到两条争议，先处理李四的，是否所有相关争议都变已处理？**

建议：**是，但前提是系统处理的对象不是“李四这张争议单”，而是整个冲突事件。**

也就是说，王五点开李四的争议单时，页面应该自动汇总同一个占用表单、同一时间段下所有待处理争议。王五提交裁定后：

| 对象 | 状态变化 |
|---|---|
| 冲突事件 | 待处理 → 已处理 |
| 李四争议单 | 待处理 → 已处理 |
| 孙六争议单 | 待处理 → 已处理 |
| 张三占用表单 | 根据裁定结果更新为“保留占用”或“被调整 / 被取消” |
| 最终占用人 | 更新为管理员选择的人 |

不建议让王五只处理“李四 vs 张三”，因为孙六其实也已经加入了同一时间段的争夺，单独处理会导致后面还要重复裁定。

## 六、问题②处理建议

**问题②：王五处理完李四争议后，孙六又发起争议，状态怎么处理？**

这里要看你们业务是否允许“裁定后继续争议”。建议默认规则是：

**管理员裁定后，该时间段进入“已裁定 / 已锁定”状态，普通用户不能再对同一塔吊同一时间段发起争议。**

这样最清晰：

| 场景 | 建议处理 |
|---|---|
| 孙六在王五裁定前发起争议 | 合并进同一个冲突事件，一起处理 |
| 孙六在王五裁定后发起争议 | 不允许发起普通争议，提示“该时段已由管理员裁定，如需调整请联系管理员” |
| 业务确实允许二次争议 | 新建“二次争议 / 申诉单”，不要和第一次争议混在一起 |

如果业务一定要允许孙六后续再争，可以这样设计：

| 单据 | 状态 |
|---|---|
| 李四争议单 D-RWS-20260704171345 | 已处理 |
| 第一次冲突事件 | 已处理 |
| 孙六新争议单 D-RWS-20260704171525 | 待处理 |
| 第二次冲突事件 | 待处理 |

但这里要注意：这会带来一个产品风险，管理员前一次裁定可能被不断推翻，现场排班会不稳定。所以除非业务强要求，否则不建议开放普通二次争议。

## 七、问题③处理建议

**问题③：王五和周七同时处理，结果不一致，以谁为准？**

不建议以“最后提交时间”为准。

正确做法应该是：**同一冲突事件只能被成功处理一次，先提交成功的人生效，后提交的人失败并提示刷新。**

比如：

| 操作顺序 | 系统结果 |
|---|---|
| 王五先提交，选择张三 | 提交成功，最终归属张三 |
| 周七后提交，选择孙六 | 提交失败，提示“该争议已由王五于 17:20 处理，请刷新查看结果” |

如果用“最后提交为准”，会有几个问题：

1. 管理员之间互相覆盖，责任不清晰。
2. 用户看到的结果可能反复变化。
3. 审计日志难解释，尤其是工地调度、安全责任场景。
4. 现场人员可能已经按第一次结果执行，后面又被覆盖，风险很高。

更稳妥的机制是：

| 机制 | 建议 |
|---|---|
| 待办领取 | 管理员打开详情后可“领取处理”，领取后其他管理员只读 |
| 并发提交 | 如果未做领取，也必须后端校验“是否已处理” |
| 状态校验 | 只有“待处理”的冲突事件可以提交裁定 |
| 操作日志 | 记录处理人、处理时间、处理结果、处理意见 |
| 后提交处理 | 提交失败并提示刷新 |

## 八、问题④处理建议

**问题④：李四发起争议后，张三把原本的占用单取消了，这里如何处理？**

建议不要简单地“禁止张三取消”，也不要直接把张三的占用单改成李四。更稳的规则是：

**张三可以取消，但取消后只是表示张三退出该时间段竞争；系统需要根据剩余参与人数量决定后续处理。**

也就是说，张三取消的动作影响的是「参与人资格」，不是直接决定最终归属。

### 推荐主规则

| 场景 | 建议处理 |
|---|---|
| 张三占用单无争议 | 张三可正常取消，占用释放 |
| 张三占用单已有争议，且只有李四一个争议人 | 可自动将该时段转给李四，或生成“待确认占用”给李四 |
| 张三占用单已有争议，且李四、孙六等多人参与 | 张三退出争议组，管理员只在李四、孙六之间裁定 |
| 张三取消时管理员已开始处理 | 需要刷新争议详情，重新展示最新参与人 |
| 张三取消后无人争议 | 争议关闭，占用释放 |

更建议采用这套规则：

**原占用人可以取消；取消后，系统将其从冲突事件候选人中移除。**

然后分两种情况：

| 剩余候选人数量 | 系统处理 |
|---|---|
| 0 人 | 关闭争议，占用释放 |
| 1 人 | 可自动归属给唯一候选人，或进入“待管理员确认” |
| 2 人及以上 | 继续保留争议待办，由管理员在剩余候选人中裁定 |

在你的例子里：

张三原本占用塔吊 A 12:00-13:00，李四发起争议，孙六也发起争议。此时参与人是：

| 参与人 | 身份 |
|---|---|
| 张三 | 原占用人 |
| 李四 | 争议人 |
| 孙六 | 争议人 |

如果王五处理前，张三取消原占用单，那么参与人变成：

| 参与人 | 身份 |
|---|---|
| 李四 | 争议人 |
| 孙六 | 争议人 |

这时管理员不需要再在张三、李四、孙六里选，而是只在**李四、孙六**之间裁定。

### 不建议直接把张三的占用改成李四

原因是：李四只是第一个争议人，不一定天然优先于孙六。尤其当孙六也已经参与争议时，直接给李四会对孙六不公平，也会让管理员介入机制失去意义。

### 也不建议完全禁止张三取消

因为张三确实可能已经不需要塔吊了。如果强制不让取消，会让资源继续被一个已放弃的人占住，反而影响现场调度。

### 原型里的推荐文案

张三点击取消时，如果存在待处理争议，可以弹窗提示：

> 当前占用时段存在待处理争议，取消后你将退出该时段占用竞争。若仍有其他争议人，系统将继续由管理员裁定最终归属。是否确认取消？

按钮：

- 确认取消
- 暂不取消

### 状态变化建议

| 对象 | 原状态 | 操作后状态 |
|---|---|---|
| 张三占用单 | 争议中 | 已取消 / 已退出争议 |
| 李四争议单 | 待处理 | 待处理 |
| 孙六争议单 | 待处理 | 待处理 |
| 冲突事件 | 待处理 | 继续待处理 |
| 管理员待办 | 待处理 | 待处理，参与人更新 |

## 九、推荐状态设计

| 对象 | 状态 |
|---|---|
| 占用表单 | 待审批、已占用、争议中、已裁定、已取消、已改派 |
| 争议单 | 待合并 / 待处理、已处理、已取消、已失效 |
| 冲突事件 | 待处理、已处理 |
| 管理员待办 | 待处理、处理中、已完成、已失效 |

原型里建议把列表命名为：**塔吊占用争议待办**，列表按「冲突事件」展示，不按单个争议单展示。

字段可以这样放：

| 字段 | 示例 |
|---|---|
| 冲突编号 | C-RWS-20260704171525 |
| 关联占用单 | TC-RWS-20260703154523 |
| 塔吊 | 塔吊 A |
| 占用时间 | 2026-07-05 12:00-13:00 |
| 原占用人 | 张三 |
| 争议人数 | 2 |
| 参与人员 | 张三、李四、孙六 |
| 当前状态 | 待处理 |
| 最早争议时间 | 2026-07-04 17:13 |
| 处理人 | 王五 |
| 处理结果 | 最终归属：张三 |

## 十、一句话结论

**争议单可以有多张，但管理员待办和最终裁定应该按“同一占用冲突事件”聚合处理；同一冲突只允许一次有效裁定，后续并发提交必须失败；原占用人取消时只代表退出竞争，如果剩余还有多人争议，管理员继续在剩余人员中裁定。**
